oracle之约束

经典语录---活着不是硬道理，活着并硬着才是道理。       3/30/2016 11:08:11 PM

约束

not null -----------不为空
unique  -----------可以有一个
paimary key-------唯一且不为空

check

1. 在新建表时
2. 后期可以添加

约束分为列级的和表级

约束可以之后来定义否则系统会给用户起一个默认的现在没什么影响但是在之后删除时就比较麻烦。

unique

  • 可以又很多列
  • primarykey 的定义一个表中只可以 有一个主键 但是可以有多列。
    • 一般的格式：
      constraint emp_emp_id primary key （employee_id，employee_name）

foreign key
定义主外键（必考）

constraint emp_dept_id foreign key （department_id） references departments(department_id)
1. 顺便如果有两个是 写两个
2. foreign key：在表级指定子表中的列
   references ：标示在父表中的列
   on   delete  cascade   当父表中的列被删除时，子表中相对于的列也别删除
   on   delete  set null  子表中相应的列置空。

check

- 出现currval，nextavl，level，和rownum违例
- 使用sysdate，uid，user，userenv函数 在查询中涉及到其他列的值
- alter table  添加和删除约束，但是不能修改约束。
- 有效化和无效化约束
- 添加not null 约束 要使用modify语句
    - alter table table
    - add  constraint constraint type （clumn）；

查询数据字典视图 

USER_CONS_COLUMNS
SELECT  constraint_name, column_name
FROM  user_cons_columns
WHERE  table_name = 'EMPLOYEES';

视图：

  • 控制数据访问
  • 简化查询（查询的速度快）
  • 数据独立性
  • 避免重复访问相同的数据

对于demo删除的改进

3/29/2016 9:34:30 PM

-问题描述

- 上次在进入用户界面进行删除，把所有的用户都删除了，页面并没有变化。
- 而且还是在那个页面上并且在页面上的缓存用户名。
- 显然很不合理
`request.setCharacterEncoding("UTF-8");
    String idString = request.getParameter("id");
    String temp_msg = "删除成功";
    try {
        int id = Integer.parseInt(idString);
        UserinfoDao userinfoDao = new UserinfoDaoImpl();
        boolean flag = userinfoDao.deleteUserinfoById(id);
        if (!flag) {
            temp_msg = "删除失败";
        }
    } catch (Exception e) {
        e.printStackTrace();
        temp_msg = "用户ID丢失或无效";
    }
    request.setAttribute("temp_msg", temp_msg);
    request.getRequestDispatcher("UserinfoListServlet").forward(request, response);`

对于delete的处理页面思考一下，也没有问题，根据跳转应该是到了userinfolistservlet的处理页面，之后就是关键的一个细节。

准对来自数据库的查询来进行判断

UserinfoDao userinfoDao = new UserinfoDaoImpl();
List<Userinfo> userinfoList = userinfoDao.fetchUserinfoList();
if ***(userinfoList == null )*** {
    request.setAttribute("temp_msg", "无用户信息，请注册！！！~");
    request.getRequestDispatcher("register.jsp").forward(request, response);
} else {
    request.setAttribute("userinfoList",userinfoList);
    request.getRequestDispatcher("userinfoList.jsp").forward(request, response);

}

对于(userinfoList == null ) 的判断本身有问题，改进之后

UserinfoDao userinfoDao = new UserinfoDaoImpl();
List<Userinfo> userinfoList = userinfoDao.fetchUserinfoList();
if (**userinfoList == null||userinfoList.size()<1**) {
    request.setAttribute("temp_msg", "无用户信息，请注册！！！~");
    request.getRequestDispatcher("register.jsp").forward(request, response);
} else {
    request.setAttribute("userinfoList",userinfoList);
    request.getRequestDispatcher("userinfoList.jsp").forward(request, response);

}

userinfoList.size()<1的判断关键。然后逻辑就可以正确。
